# Quality Gate - Custodian Workflow

Pre-merge composite check that runs health, pattern, and migration audits, then produces a unified PASS / FAIL / CONCERNS verdict.

---

## Step 1: Run Repo Health Check

Execute the checks defined in the `repo-health-check` workflow instructions:

1. File inventory counts
2. Model registration completeness
3. Router registration completeness
4. Duplicate detection (migrations, classes, prefixes)
5. Structure verification

Record each check result as PASS or FAIL. Collect all findings into a `health_results` section.

---

## Step 2: Run Pattern Audit

Execute the checks defined in the `pattern-audit` workflow instructions:

1. Tenant isolation audit (every query filters by `tenant_id`)
2. SQL injection prevention (no f-string SQL)
3. API endpoint RBAC coverage
4. Type hint compliance
5. Frontend pattern compliance
6. Model inheritance verification

Record each check result as PASS, WARNING, or FAIL. Collect all findings into a `pattern_results` section.

---

## Step 3: Run Migration Audit

Execute the checks defined in the `migration-audit` workflow instructions:

1. Duplicate migration number detection
2. Revision chain validation (no broken links, no orphans, single head)
3. Tenant isolation in new tables
4. Migration content quality
5. Model-migration alignment

Record each check result as PASS, WARNING, or FAIL. Collect all findings into a `migration_results` section.

---

## Step 4: Compute Verdict

Apply the following verdict logic:

### FAIL (blocks merge)
Any of the following triggers an immediate FAIL:
- Missing tenant_id filter in a database query (tenant isolation violation)
- f-string SQL detected (SQL injection risk)
- Broken Alembic revision chain (migrations will not run)
- Duplicate migration numbers (Alembic will crash)
- Model file exists but is not registered in `__init__.py`
- Router file exists but is not registered in `main.py`

### CONCERNS (merge at discretion)
Any of the following triggers CONCERNS if no FAIL conditions exist:
- Endpoints missing RBAC dependencies
- Functions missing type hints
- Migrations with empty `downgrade()` functions
- Multiple Alembic heads (unmerged branches)
- Models registered but not used by any service
- Frontend components with no imports

### PASS
All checks pass with zero FAIL and zero CONCERNS findings.

---

## Step 5: Produce Report

Output a structured report to `{output_folder}/custodian/quality-gate.md` with:

### Header
```
============================================
  QUALITY GATE VERDICT:  [PASS|CONCERNS|FAIL]
============================================
  Branch:    {current git branch}
  Date:      {current date}
  Commit:    {current HEAD short hash}
============================================
```

### Sections

1. **Verdict Summary** - One-line explanation of the verdict.
2. **Blocking Issues (FAIL)** - Numbered list of critical findings. Empty if PASS.
3. **Concerns (WARNING)** - Numbered list of non-blocking findings. Empty if PASS.
4. **Health Check Results** - Summary table from Step 1.
5. **Pattern Audit Results** - Summary table from Step 2.
6. **Migration Audit Results** - Summary table from Step 3.
7. **Score Breakdown**
   - Health score: X/Y checks passed
   - Pattern score: X/Y checks passed
   - Migration score: X/Y checks passed
   - Overall: X/Y total checks passed (percentage)

### Footer
```
---
Generated by: Code Custodian (Sentinel) - Quality Gate Workflow
```

---

## Step 6: Present Results

After writing the report file, output the verdict header and any blocking issues directly to the user. If the verdict is CONCERNS or FAIL, list the top 5 most important items to address before merge.
